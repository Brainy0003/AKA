First we have our Using Directives added by Visual C#, such as: System and System.Windows.Form. These allow us to access the Windows Operating System and tell it that we want to create a Form window.

Next we have a Namespace Declaration
Tutorial writers seem to waste pages and pages trying to explain what Namespaces are and never seem to get anywhere. It's quite simple really:
Let's say you have two programmers working for the same company. Bob over in Dept. A writes some code and stores the company's customer profiles in a variable called X. Meanwhile over in Dept. B, John the programmer is implementing another piece of code which stores the company's laundry list in another variable called X. Of course you can imagine the merry mixups that occur when the two parts of the program are brought together and one day the customer database gets overwritten with the dirty towels list.
What the Namespaces allow you to do is put Bob's code in one container like so:

namespace BobsCode
{
int x = CustomerDatabase ;
etc...
}

and John's code in it's own container:

namespace JohnsCode
{
int x = DirtyLaundry ;
etc...
}

Now you can put both pieces of code on the same page and the variable names won't clash.
One suggested format is YourCompanyName.YourProjectName.

You can have Namespaces spread over more than one file. Flip back and forth between Form1.cs and Program.cs and observe that they both share the same Namespace. This allows you to organize a big project into smaller separate files. If they are within the same Namespace, to the computer it's as if they were together on the same page.

In Form1.cs, click after the end of the line that says InitializeComponent(); and hit Enter to create a new line. Type sys
a window should pop open with the word System highlighted. Simply hit Enter and Visual C# completes the rest of the word for you.

Now type a period. A list of available namespaces in the System namespace appears. You can use the Arrow Keys to scroll up and down this list and explore the available namespaces.

In this case, we have created a two level namespace. First Tutorials, then UsingAForm. Try typing tu in the code window. Note that the popup window has jumped to Tutorials in the list. Simply hit Enter and Auto Completion types the rest of the word for you. Now type a period and UsingAForm shows as the only Namespace available in the Tutorials Namespace. Type a u or hit the down arrow and hit Enter and it will write it for you. If you put a dot after UsingAForm it will show you the class we have created: Form1 in the list.

OK let's delete the Tutorials.UsingAForm.etc... line we were just playing around with.

The name Form1 is just a generic name put out there and we don't necessarily want to keep it that way.
After public partial class, double-click on Form1 and change it to MyForm. Note that some little red squiggly lines appear under some of the names indicating that something has gone wonky. Visual C# has some built in features which will automatically update any immediate discrepencies.
Observe that if you hold your mouse over the little red squiggly bit, a menu tab pops open from which you can now select the option to update any dependencies you may have overlooked.

Select Rename 'Form1' to 'MyForm' and observe that the name has been updated in more than one place in your code. Any necessary adjustments are also made to files which are normally hidden.

To illustrate, I want you to hold Ctrl Z until you see that MyForm has changed back to Form1. OK, Everything is back the way it was. You can hit the F5 key to see that the application is indeed not broken.
OK, let's close that and double click the word Form1 to select it.
Hold Ctrl H. The Find and Replace window opens.
In the Replace With field, enter MyForm. Under Look In, make sure you select Current Project and click Replace All. A Window pops up indicating that 5 ocurrences have been changed. Hit Enter or click OK.
Observe that a new page has also opened in the Editor Window now called Form1.Designer.cs
This is one of the pages of code which Visual C# creates for you automatically which handles a bunch of routine Windows crap. At first glance however, you may not see that the word MyForm has actually been updated in the code file.
If you scroll down to the bottom, you'll see a line which says private void InitializeComponent() .....
if you click on the + sign to the left of that, you will see that it expands the text. This is another nifty feature of Visual C# which allows you to close groups of code up into nice manageable chunks.
OK let's close Form1.Designer.cs to get it out of our way by clicking on the little black X, and be sure that Form1.cs is the active window by clicking on the Form1.cs Tab at the top.

Update the code as follows:

using System;
       using System.Collections.Generic;
       using System.ComponentModel;
       using System.Data;
       using System.Drawing;
       using System.Text;
       using System.Windows.Forms;       
namespace Tutorials.UsingAForm
{
    public partial class MyForm : Form
    {
         public MyForm()
         {
         InitializeComponent();
         }
 protected override void OnPaint( PaintEventArgs e )
         {
         e.Graphics.DrawString ( "Hello Windows", this.Font,
         Brushes.Black, 10, 10);
         }
   }
}

Hit the F5 key to see your creation.

Upon running this code, we will get a Form window. Near the top corner of that Window it should say "Hello Windows".

Let's have a little fun with the colors. Double-click on the word Black and hit the Backspace key 2 times to delete the ColorName and the period before it. Now type another period in it's place. A window appears with a list of available color presets. Scroll down to a new color name and hit Enter.
Try changing the numbers to 150, 50
Observe how increasing the first number moves the text to the right, while increasing the second number moves it further down.
